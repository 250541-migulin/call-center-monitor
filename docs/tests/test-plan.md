# Test Plan  
**Система мониторинга технической поддержки**

## 1. Введение
Настоящий тест-план описывает подход к тестированию веб-приложения «Система мониторинга технической поддержки» в рамках выполнения лабораторной работы №6.  
Цель — подтвердить соответствие реализованной системы требованиям, заданным в SRS и Use Case’ах (UC1–UC9).

## 2. Объект тестирования
Тестируемый продукт — веб-приложение, включающее три пользовательских интерфейса:
- **Оператор**: регистрация обращений, заполнение карточки клиента.
- **Руководитель**: просмотр статистики, фильтрация, экспорт данных, анализ звонков вне графика.
- **Администратор**: управление пользователями, настройка рабочего времени.

Ключевые компоненты:
- Frontend (веб-интерфейс)
- Backend API (сервисы: InteractionService, AnalyticsService, ExportService)
- База данных (PostgreSQL)

## 3. Аспекты тестирования

### 3.1 Функциональные требования (оценка по SRS)
Будут протестированы все функциональные требования из SRS:
- Регистрация обращения (UC3)
- Заполнение карточки клиента (UC4)
- Управление пользователями (UC1)
- Настройка рабочего времени (UC2)
- Просмотр истории, статистики, фильтрация, экспорт (UC5–UC8)
- Анализ звонков вне рабочего времени (UC9)

### 3.2 Нефункциональные требования
- **Юзабилити**: соответствие мокапам (`/docs/mocks/`)
- **Надёжность**: корректная обработка ошибок (валидация, сетевые сбои)
- **Интернационализация**: поддержка кириллицы в CSV-отчётах (UTF-8)

## 4. Подход к тестированию

### 4.1 Use Case-тестирование
Основной метод — **тестирование на основе Use Case’ов**.  
Для каждого UC разработаны:
- **Позитивные тест-кейсы** (основной поток событий)
- **Негативные тест-кейсы** (альтернативные потоки: A1, A2 и т.д.)

### 4.2 Типы тестов
- **Ручное функциональное тестирование** через веб-интерфейс
- **Проверка соответствия мокапам**
- **Валидация форм** (обязательные поля, формат времени, уникальность логина)
- **Интеграционные проверки** (сохранение → отображение в истории)

### 4.3 Инструменты
- Браузер: Google Chrome (актуальная версия)
- Среда: локальный запуск (dev-сервер)
- Документация: GitHub (Markdown)

## 5. Критерии начала тестирования
- Завершено проектирование (SRS, Use Cases, UML-диаграммы — ЛР №2, №3–4)
- Реализованы ключевые функции (ЛР №5)
- Развернута тестовая среда

## 6. Критерии завершения тестирования
- Все позитивные тест-кейсы пройдены
- Все критические дефекты (блокирующие UC) исправлены
- Покрытие требований: **100%** (каждое требование из SRS покрыто хотя бы одним тест-кейсом)

## 7. Риски
- Несоответствие интерфейса мокапам
- Отсутствие валидации на стороне клиента
- Ошибки в расчёте «внерабочих» звонков (требование 3.1.11)
- Некорректная кодировка CSV (кириллица не читается в Excel)
